# ARCHITECTURE.md

本仓库只为一件事服务：

让 AI 可以在一个长期可演化、不会失控的项目里持续产出代码，而不是把仓库推成屎山。

## 第一性原理

AI 驱动的项目要长期健康，关键不在“模型一次回答得多好”，而在于仓库是否天然支持下面这几个动作：

1. 明确目标
2. 按需获取上下文
3. 在清晰边界内修改
4. 用外部信号验证结果
5. 把有效状态保留下来

所以仓库结构必须优先服务于 Agent 的执行闭环，而不是服务于文档观感。

## 最小结构

本仓库只保留最必要的几个顶层组件：

- `soul.md`：最小灵魂约束，永远保持精简
- `AGENT.md`：Agent 首先读取的入口和工作闭环
- `RULES.md`：默认约束、协作规则、验证要求
- `ARCHITECTURE.md`：系统边界和目录职责
- `SPECS/`：目标、需求、任务说明
- `src/`：业务代码
- `tests/`：验证代码
- `TASKS/`：当前和历史任务记录
- `MEMORY/`：跨任务保留的决策与经验
- `WORKLOG/`：按天记录的项目推进日志
- `.learnings/`：按天沉淀的复盘与反思
- `ARTIFACTS/`：运行证据、产物、截图、日志

## 为什么这样分

### `SPECS/`

目标和约束应该与实现分离。

Agent 写代码前，应该先知道要达到什么状态，而不是先钻进 `src/` 里猜。

### `src/`

所有业务实现都放在这里。

不要把业务代码散落到仓库其他位置。

### `tests/`

验证必须和实现并列存在。

一个没有验证层的 AI 项目，后期几乎必然失控。

### `TASKS/`

任务状态不能只留在聊天上下文里。

每个正在推进的重要任务，都应该有一个独立文件，记录：

- 目标状态
- 当前约束
- 验收信号
- 已做动作
- 下一步

### `MEMORY/`

跨任务有效的信息要沉淀，但不要散成几十个小文件。

只保留两类高价值记忆：

- 决策
- 经验

### `WORKLOG/`

每天一份工作日志。

目的不是写周报，而是让项目推进痕迹清晰可追。

每天至少记录：

- 今天推进了什么
- 遇到了什么问题
- 做了什么决定
- 明天的下一步是什么

### `.learnings/`

这里放每天的反思，不放流水账。

目的不是记事，而是让 Agent 在第二天变得更聪明。

每天至少记录：

- 今天最有效的做法
- 今天最浪费的做法
- 哪个判断以后要默认提前做
- 哪个风险需要进入默认约束

### `ARTIFACTS/`

证据和结果不要混进源码目录。

日志、截图、生成产物、临时分析结果，都应放到这里。

## 默认执行路径

Agent 进入仓库后，默认遵循这个顺序：

1. 读 `AGENT.md`
2. 读 `RULES.md`
3. 找当前任务对应的 `TASKS/` 文件
4. 看当天 `WORKLOG/` 和最近 `.learnings/`
5. 如仍不清楚边界，再读 `ARCHITECTURE.md`
6. 然后再进入 `SPECS/`、`src/`、`tests/` 的最小必要范围

## 目录设计原则

- 少层级，少跳转
- 一类信息只放一个地方
- 重要状态落文件，不依赖聊天记忆
- 代码、目标、验证、证据彼此分离
- 如果一个目录长期没有承担清晰职责，就删掉

## 最终标准

好的结构不是“文档很多”，而是让 Agent 能稳定回答这五个问题：

1. 现在目标是什么
2. 下一步该读什么
3. 代码该写到哪里
4. 改完怎么验证
5. 这次学到的东西存到哪里
