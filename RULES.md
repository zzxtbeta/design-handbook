# RULES.md

这个文件只定义默认约束。

不要把口号写进来，只保留真正会影响执行结果的规则。

## 默认规则

1. 先明确目标，再改代码。
2. 先找验收信号，再声明完成。
3. 一次只解决一个清晰问题，避免同时改很多边界。
4. 优先读取最小必要上下文，不默认扫描整个仓库。
5. 重要状态写入文件，不只留在对话里。
6. 连续两次同类失败后，不再盲试，必须换策略或补验证。
7. 没有证据，不算完成。
8. 默认由 Agent 承担 PM 推进职责，只有关键方向才询问用户。
9. 每天结束前必须补齐工作日志与反思。

## 上下文规则

- 先读 `AGENT.md`
- 再读 `RULES.md`
- 再读当前任务文件
- 再看当天工作日志和最近反思
- 然后只读当前动作需要的最小文件集合
- 如果局部信息不足，再读 `ARCHITECTURE.md`

## 任务规则

每个重要任务都应该在 `TASKS/` 中有记录。

任务文件至少包含：

- 目标状态
- 约束
- 验收信号
- 当前结论
- 下一步

## 日志规则

- `WORKLOG/` 每天一份日志
- `.learnings/` 每天一份反思
- 没有日志和反思，这一天就不算真正闭环

## 验证规则

优先使用更强的验证方式：

1. 确定性检查
2. 自动化测试
3. 运行时检查
4. 产物对比
5. 人工确认

如果一个改动没有任何可靠验证方式，应先承认这个缺口，而不是假装已经完成。

## 协作规则

- 一个任务切片同一时间只应有一个 owner
- 交接必须带状态、范围、证据和下一步
- 多 Agent 只有在角色价值已经被证明时才值得拆分

## Skills / Agents 最小原则

### Skills

只有面向高价值问题的端到端完整解法，才值得沉淀为 Skill。

局部技巧、零散 prompt、临时动作，不进入 Skill。

### Agents

Agent 是角色化执行接口，不是人格设定。

只有某类角色在真实任务中反复证明有价值，才值得被固化。
