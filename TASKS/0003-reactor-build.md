# 0003 Reactor Build

## 目标状态

在不破坏现有情绪板设计语言的前提下，完成创作反应堆首版开发规划，并按小版本推进到可用状态。

## 约束

- 审美板现有主体验不重做，只补必要小功能
- 创作反应堆独立成板块，不和审美板混池
- 创作反应堆按天展开
- 默认录入不调用 AI
- AI 只在用户主动整理时介入
- 删除动作必须支持降熵，不只是直接删
- 前端每个小版本都先给用户过方向
- 每次给用户验前端版本前，先把当前版本同步到 GitHub
- 每个大版本收口时，打清晰 tag
- 后端、数据边界、实现规范由我持续兜住
- 每个版本都必须能被验证，不留半截状态

## 验收信号

- 存在双板块入口，但视觉语言统一
- 审美板现有 UI 不被破坏
- 创作反应堆存在按天白板
- 用户可新增日记、点子、prompt、链接、长文样本
- 用户可给素材补备注和手动标签
- 用户可多选素材并触发整理动作
- AI 整理只在主动触发时出现
- 删除支持直接删、合并、归子集、降级次要碎片
- 每个阶段都有明确演示面和回退边界

## 当前结论

- 审美板方向已稳定，适合在现有基础上继续演进
- 创作反应堆新版方向已锁定为“按天展开的乱序创作白板”
- 旧版 Reactor spec 已删除
- 新版 Reactor spec 已落在 `SPECS/0005-creator-reactor-board.md`
- 当前最重要的不是继续讨论，而是按小版本把 UI 和数据边界搭出来

## 版本节奏

### V0.1 壳子接入

目标：

- 加入双板块入口
- 保持现有审美板完整可用
- 创作反应堆先有空壳页面和基础版式

用户验收重点：

- 入口位置对不对
- 视觉语言是否还是同一个产品
- 新板块有没有破坏现有情绪板气质

后端状态：

- 可先不上完整数据读写
- 允许先用本地假数据占位

### V0.2 Daily Board 录入

目标：

- 创作反应堆按天展开
- 支持新增五类素材
- 支持备注和手动标签
- 默认完全不调用 AI

用户验收重点：

- 白板感够不够自由
- 卡片展示是否轻
- 录入动作是否足够顺手

后端状态：

- 开始落真实数据模型
- API 先保证创建、读取、更新最小闭环

### V0.3 多选与整理条

目标：

- 支持多选素材
- 出现 Selection Bar
- 支持删除、合并、归子集、降级次要碎片这些动作入口

用户验收重点：

- 这些动作是否符合真实创作判断
- 页面会不会因为功能多了就变脏
- 整理动作是否比“继续堆着”更顺手

后端状态：

- 建立素材关系表达能力
- 明确合并 / 子集 / 降级的数据边界

### V0.4 主动 AI 整理

目标：

- 支持“梳理日记”
- 支持“整理选中素材”
- 支持“判断是否并入另一条”
- 支持“整理成标准说明”

用户验收重点：

- AI 介入是否自然
- 输出是否有帮助而不抢戏
- 是否真的在减熵，而不是制造更多内容

后端状态：

- 补齐异步处理状态
- 补齐失败态
- 补齐最小可回放日志

### V0.5 交付收口

目标：

- 打磨交互细节
- 修正视觉噪音
- 完成基础 QA
- 确认可持续继续开发

用户验收重点：

- 整体是否已经是一个能每天用的板块
- 还有哪些方向性不对

后端状态：

- 类型检查通过
- 构建通过
- 基础冒烟验证通过

## 责任分工

- 在每个前端小版本后做一次口头验收
- 只在方向和品味上做关键判断

### 我负责

- PM 推进
- 版本拆分
- 前后端实现
- 数据模型收口
- AI 接入边界
- 构建、类型检查、QA 验证
- 不让技术债滚成后患

## Harness 原则

每个版本都必须满足：

- 有明确输入
- 有可见输出
- 有可验证状态
- 有失败时的回退方式

具体执行上：

- 前端先做最小可演示面
- 后端同时补最小真数据链路
- 每一版结束都跑类型检查和构建
- 每一版准备验收前，先提交并推送当前分支
- 每个大版本完成后，补版本 tag
- 后续接入 gstack 做页面和交互验收

## 当前推荐执行顺序

1. 先做 V0.1，接双板块入口和 Reactor 空壳
2. 你看方向
3. 再做 V0.2，把按天白板和录入做出来
4. 你再看方向
5. 方向稳定后，我连续推进 V0.3 到 V0.5
