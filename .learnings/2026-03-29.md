# 2026-03-29

## 今天最有效的做法

- 先把仓库结构减到最小，再开始定义产品和流程
- 用 `office-hours` 先锁产品本质，再进入实现规划
- 先把 PostgreSQL 和真实 Gemini 路径跑通，再讨论更细的体验层，能更快暴露真正会卡住产品的边界
- 在兼容第三方模型网关时，直接保留用户给的 base URL 并只补兼容参数，比“替用户修正配置”更稳

## 今天最差的做法

- 前期文档层级一度偏多，容易让 Agent 多跳文件
- 同一天里不断刷新“明日下一步”，说明执行顺序还不够受一个单一北极星约束
- 在日视图排版是否要持久化还没定之前，就先把本地状态交互做复杂了

## 以后默认提前做什么

- 尽早锁定“首版不做什么”
- 尽早确认谁负责项目推进，避免把 PM 工作重新丢给用户
- 从第一天就启用工作日志和反思，而不是做了一半再补
- 在实现前先写清楚数据模型和异步流，避免边做边猜
- 发现会污染源码目录的构建产物时要立刻收口，不要等以后再清理
- 先把“假数据 UI”替换成“真实 API + 假实现”很值钱，这能更早暴露边界问题
- 模型厂商边界要尽早抽出来，越晚抽，越容易把供应商协议渗进业务代码
- 时间骨架也要尽早做真，不然整个产品会一直像假 demo
- 数据库里“这周是谁”不能靠时间戳精确比较，显式稳定 key 更可靠，也更利于 Agent 理解
- 早一点做真实运行时验证很值钱，`GET` 能过不代表 `PUT` 和异步状态流没坑
- 第三方模型网关即便叫 Gemini，也可能走 OpenAI 兼容协议，边界设计不能把厂商名和协议名绑死
- 对强视觉产品，凡是“看起来像正式能力”的前端状态，都要尽早确认是不是要落库；不确认就不要让交互成熟得太像已完成

## 要升级为规则的东西

- `soul.md` 必须始终保持极简
- 每天没有 `WORKLOG` 和 `.learnings`，就不算完成闭环
- 首版范围必须用“明确不做”来保护，不然美学产品很容易膨胀
- 构建和类型检查默认要做到“不会在源码目录留下脏产物”
- 跑通第一条真实运行时链路，比再补一层文档更值钱
- 图片存储边界应尽早真实化，否则 Data URL 很快会把接口和状态拖脏
- 对这种强视觉产品，先把“不是 dashboard”这件事做对，比先多加一层功能更重要
- 后端从假持久化切真持久化时，必须连同运行时请求一起验，不接受只靠 typecheck 判断“已完成”
- 涉及时间、周次、日历这类对象时，优先使用显式稳定标识，不把精度和时区问题留给后面
- 用户给定的模型入口参数默认视为产品约束，不擅自替换；需要做的是兼容，而不是改用户输入
- 当某个体验同时存在“数据库真状态”和“浏览器本地状态”两套来源时，必须先决策主事实来源，再继续做 UI 精修
- `lint` 和最小可运行回归脚本要尽早接线；否则每天总结时很难把“今天真的稳定了什么”说得足够硬

## 今日追加反思

- 真实模型“看起来像接好了”和“用户感知上真的接好了”是两回事，必须用不同图片做上传级回归
- 模型失败时静默回退 mock 会制造最糟糕的假象，比直接报错更伤产品判断
- 日笔记这种显式用户输入一旦进入 UI，就不能继续停在浏览器本地草稿层
- 当迁移账本和数据库现实状态漂移时，应先用幂等方式保住功能，再专门修迁移历史，不让账本问题阻断主线
- 像图片描述这种“有用但不该占空间”的信息，最好默认折叠，放在 hover / click 之后露出
- README 不是给人看热闹的，它首先是给未来的 Agent 一个低歧义入口；入口越短、启动路径越真，后面越不容易长成屎山
- 真正的“干净”不是没有 TODO，而是主事实源、启动方式、失败模式都已经清楚，剩余债被明确地压缩到最小范围
