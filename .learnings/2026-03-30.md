# 2026-03-30

## 今天最有效的做法

- 先确认问题是否真实存在，再改代码。图片删除不删磁盘文件这次就是先查清链路，再补删除逻辑
- 目录整理只删空壳，不碰真实信息密度高的结构，能快速减熵又不伤项目历史
- 上传文件按日期组织，是一种非常低成本但长期复利很高的可维护性设计
- 在 README 里把“怎么启动”和“数据到底落哪”写实，比继续加概念文档更值钱

## 今天最差的做法

- 之前把图片删除语义只做到了数据库层，没有一路贯彻到文件层，这会让系统表面干净、底层持续堆积
- 空的 `src/` 和 `tests/` 占位留得太久，会误导后续 Agent 以为这里有第二套代码入口

## 以后默认提前做什么

- 凡是有本地文件落盘的功能，第一次实现时就同步设计删除与回收路径
- 顶层目录默认只保留“现在就有明确职责”的结构
- 每次 README 改完，都顺手核对一遍真实脚本名，避免文档先漂
- 涉及上传文件时，默认让路径自带时间信息，方便审计、排查、清理

## 要升级为规则的东西

- 删除数据库记录如果对应着磁盘文件或外部资源，删除语义必须一路贯彻到底
- 空壳目录不是“以后扩展性”，而是当前歧义源；没有明确职责就删
- 项目入口文档必须回答三个问题：
  - 这是什么
  - 本地怎么启动
  - 数据存在哪里

## 今日反思

- 第一性原理看这类项目，最重要的不是“目录看起来像不像标准模板”，而是 Agent 能否快速定位事实源、入口和副作用边界
- 图片上传的真正闭环不是“能显示”，而是“能存、能找、能删、删得干净”
- 现在目录已经比昨天更干净了，但真正该补的下一层护栏还是自动化回归，而不是再加新结构
