# 2026-04-01

## 今天确认的事情

- UI 打磨如果没有和持久化、资源清理一起收口，很容易制造“表面更顺了，底层更脏了”的假进展
- 本地文件缓存一旦参与用户体验，就必须被当作正式数据层的一部分对待，而不是临时附件
- 编辑能力对 Reactor 很关键，但编辑器必须是“查看判断上下文 + 轻修改”，不能滑向完整后台表单
- 文件系统健康检查不能偷懒做平铺假设；只要上传目录支持分层，就必须递归核对引用，否则很容易得到错误告警
- 情绪板之所以更美，不是因为元素更多，而是因为视觉主次更单一；Reactor 后续要继续向“主内容更稳、操作更退后”收

## 今天踩到的点

- 挂在卡片外侧的操作按钮，看起来轻，但在复杂边界、overflow、hover 组合下非常容易被裁掉
- `jsonb meta` 这种灵活结构短期很好用，但如果不持续收住字段边界，很容易悄悄变成技术债
- 链接预览最难的不是前端渲染，而是：
  - metadata 解析鲁棒性
  - 外链封面稳定性
  - 资源生命周期管理
- 视觉参考图里的“真实感”往往不是靠大装饰，而是靠极少数细节做真：比如图钉的高光、针脚、材质感，这类东西比多一层装饰更有用

## 以后默认做法

- 只要引入“外链资源 -> 本地缓存”这条链路，就默认补上删除时的反向清理
- 只要一个界面里出现布局 localStorage，就默认在读入时做坏数据清理，在写入时做存活对象裁剪
- 新增编辑入口时，优先先问：
  - 用户此刻最想确认什么上下文
  - 用户此刻最常改哪两个字段
  - 其他字段是否真的值得露出来
- 只要做数据库 / 文件系统健康检查，就默认跑三件事：
  - API health
  - 表计数
  - 上传引用递归核对

## 值得保留的决定

- 默认 tag 先从素材类型开始，而不是一开始就让用户自己发明分类法
- `Why keep it` 这类字段适合预设短句，因为它本质上更像创作判断动作，不像正式笔记
- 每个小版本结束前都同步 GitHub，并留下日志，这让节奏更稳，也让项目健康更容易被看见
- 情绪板关键词展开改成“一体式 pill + 内置复制图标”是对的，复制动作属于标签本体，不应该被拆成旁边一个独立工具

## 当前判断

- 项目当前没有明显结构腐化，问题更多是“下一阶段要不要及时切入真正有价值的交互”
- 当前最该守住的不是视觉，而是工程节奏：继续用小版本推进，但每一轮都把链路和健康一起收口
- 当前 Postgres / 文件系统 / API 三层状态是健康的，所以接下来可以把更多注意力放回交互与审美，而不是底层补洞
