## 关键决定

- 产品定义：视觉手帐 + 设计语言引擎
- 首版范围：只做最强闭环，不做分析面板和长期词典页
- 仓库结构：全部保留在 `handbook` 内统一管理
- 工程结构：`apps/web + apps/api + packages/db`
- 数据模型：显式 `weeks / entries / entry_terms / week_notes`
- AI 处理：数据库驱动轻异步流，不上独立队列
- 协作方式：Agent 默认负责 PM 推进，用户主要做验收
- 前端构建策略：只做类型检查，不在源码旁生成构建产物
- 前端从本地假数据切换为真实 API 读取
- 后端首版先使用 in-memory store 承接真实交互，再逐步替换为数据库
- AI 边界：业务层不绑定单一模型厂商
- 图片边界：entry 记录 URL，不长期保存 Data URL
- 时间边界：周视图不再是假日期，后端按 offset 生成真实周元信息
- 数据库环境：本地开发默认使用 Docker Postgres，而不是隐式依赖本机安装
- 周标识：数据库内部使用显式稳定 key，而不是依赖时间戳精确比较
- Gemini 接入：优先兼容用户提供的 OpenAI 兼容网关形态，不擅自改 base URL
- 设计洞察数据不只保留术语，还要保留一句可检索的 `promptSummary`，方便后续回看、搜索和二次生成
- 当前更复杂的版式交互先允许存在于前端本地状态中，但这只是过渡，不应被误判为已完成产品能力

## 阻塞 / 风险

- 首版最容易因“想把美学做满”而发生 scope creep
- AI 提词体验如果等待感和失败状态处理不好，会直接伤害产品气质
- 目前前端只是静态骨架，真实数据流和上传链路还没有接上
- 当前图片仍走 Data URL，本地开发快但后续必须换成真实存储边界
- 当前数据库还没接上，持久化仍然只完成了图片文件层
- 现在的视觉方向已经更像手帐，但还没有达到参考图那种“散落感”和“密度”
- 现在后端已真落库，但前端视觉密度和纸面散落感仍明显落后于参考图
- AI provider 已抽象，但真实 Gemini 实调用和失败重试策略还没接完
- 当前 Gemini 网关对 SVG 输入不稳定或不支持，前端实际上传应继续以 PNG/JPEG 为主
- 当前周笔记已持久化到数据库，但日视图排版和日备注仍落在 `localStorage`；如果继续叠加这层交互而不先决定是否要落库，用户会遇到“内容是真的、版式是假的”的分裂体验
- `test` / `lint` 仍未接线，当前项目健康主要靠 typecheck 和人工跑通链路兜底

## 浪费动作
